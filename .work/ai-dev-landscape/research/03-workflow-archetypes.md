# Pass 3: Workflow Archetypes -- How Power Users Actually Work

## Questions Investigated

1. Spec-driven workflows: Harper Reed, Boris Tane, Addy Osmani variations and tradeoffs
2. Parallel-agent coordination: Simon Willison, Claude Squad, native agent teams, agent_farm
3. Continuous/autonomous patterns: Ralph Wiggum loops, Geoffrey Huntley, agent_farm unattended operation
4. High-volume human-in-the-loop: Boris Cherny (100+ PRs/week), Shrivu Shankar (billions of tokens/month)
5. Failure modes: reliability ceilings, context degradation, error compounding, and effective mitigations

## Findings

### 1. Spec-Driven Workflows

Three major variants have emerged, each emphasizing a different phase of the spec-plan-execute pipeline.

#### Harper Reed: Spec-First with Prompt Plans

**Source:** [harper.blog/2025/02/16/my-llm-codegen-workflow-atm](https://harper.blog/2025/02/16/my-llm-codegen-workflow-atm/), [harper.blog/2025/05/08/basic-claude-code](https://harper.blog/2025/05/08/basic-claude-code/)

Reed's workflow generates three artifacts:
- `spec.md` -- brainstormed via conversational LLM ("Ask me one question at a time so we can develop a thorough spec")
- `prompt_plan.md` -- generated by a reasoning model (o3), containing sequential prompts for implementation
- `todo.md` -- lower-level task checklist

The entire planning phase takes ~15 minutes. He calls this "a waterfall in 15 minutes." After Claude Code launched, he migrated from Aider and now uses a meta-prompt that reads `prompt_plan.md`, implements each incomplete task, runs tests, commits, and pauses for review.

**Key insight:** "The robots LOVE TDD. Seriously. They eat it up." He positions TDD as the most effective counter to hallucination and scope drift, reporting greenfield projects complete in 30-45 minutes.

**Tradeoffs:** Self-described as "largely a solo endeavor" -- lacks multiplayer collaboration. The prompt_plan.md approach is optimized for greenfield; Reed doesn't recommend it for established codebases.

#### Boris Tane: Research-Plan-Annotate-Execute

**Source:** [boristane.com/blog/how-i-use-claude-code](https://boristane.com/blog/how-i-use-claude-code/)

Tane's core principle: "never let Claude write code until you've reviewed and approved a written plan." His three-phase workflow:

1. **Research:** Deep codebase analysis written into `research.md`. Prompts emphasize "deeply," "in great details," "intricacies" to prevent surface-level skimming.
2. **Planning:** Claude generates `plan.md` with explanations, code snippets, file paths, and tradeoffs. Uses custom markdown files instead of Claude Code's built-in Plan Mode for "full control and persistence."
3. **Annotation cycles (1-6 iterations):** Tane adds inline notes directly into `plan.md` -- domain knowledge, rejected approaches, business constraints, corrections. Claude updates the plan without implementing. The markdown file becomes "shared mutable state."
4. **Implementation:** A single reusable prompt: "implement it all. when you're done with a task or phase, mark it as completed in the plan document. do not stop until all tasks and phases are completed."

**Critical innovation:** The annotation cycle. Instead of explaining holistically through conversation, Tane points to exact locations in the plan needing revision. This concentrates human judgment upfront rather than distributing it across implementation feedback loops.

**Tradeoffs:** Runs in a single long session (research through implementation). Reports no significant degradation, attributing this to auto-compaction and the persistent `plan.md` surviving compression. Actively reverts bad approaches rather than patching them -- "git reverts and re-scoping" over incremental fixes.

#### Addy Osmani: Chunked Implementation with Granular Commits

**Source:** [addyosmani.com/blog/ai-coding-workflow](https://addyosmani.com/blog/ai-coding-workflow/)

Osmani's approach emphasizes breaking projects into "bite-sized tasks or milestones" with ultra-fine-grained version control:
- Commit after each small task, treating commits as "save points in a game"
- Each iteration carries forward context, incrementally adding features
- Start in Plan Mode (Shift+Tab), let the agent draft a spec while exploring code

**Key distinction from Reed and Tane:** Osmani focuses on the execution granularity rather than the planning artifact. Where Reed generates a prompt plan and Tane annotates a plan document, Osmani emphasizes the "one task at a time" discipline with commit-per-task as the safety net.

**Tradeoffs:** Less prescriptive about planning format, more about execution discipline. Reports that "planning upfront pays off enormously" but positions the developer as "director of the show."

#### Jesse Vincent: Superpowers Framework (Architect/Implementer + TDD)

**Source:** [blog.fsck.com/2025/10/09/superpowers](https://blog.fsck.com/2025/10/09/superpowers/), [github.com/obra/superpowers](https://github.com/obra/superpowers)

Vincent's framework (40,000+ GitHub stars) enforces RED/GREEN TDD: write a failing test, implement only enough code to pass, then advance. His workflow evolved from an explicit architect/implementer split (two Claude sessions) to a single lead that dispatches tasks to subagents one-by-one with code review between each.

**Key innovation:** Skills are tested using "pressure scenarios" based on Cialdini's persuasion principles -- scarcity, authority, commitment. Vincent found that "persuasion principles work on LLMs," making testing with realistic pressure more effective than quiz-style validation.

#### Freek Van der Herten: Custom Agents per Task Type

**Source:** [freek.dev/3026-my-claude-code-setup](https://freek.dev/3026-my-claude-code-setup)

Van der Herten maintains four custom agents: `laravel-simplifier` (Opus, code refinement), `laravel-debugger` (Sonnet, bug tracking), `laravel-feature-builder` (Opus, new features), and `task-planner` (Opus, task decomposition). Uses 40+ skills and a custom status line color-coded by context window usage (green <40%, yellow 40-59%, red 60%+).

**Key insight:** The status line provides constant visual feedback about context health, prompting fresh sessions before degradation.

#### Comparative Analysis

| Practitioner | Planning artifact | Human control point | Execution model | Best for |
|---|---|---|---|---|
| Harper Reed | spec.md + prompt_plan.md | Pre-generated prompts | Sequential prompt execution | Greenfield projects |
| Boris Tane | research.md + plan.md | Annotation cycles (1-6x) | Single long session | Brownfield with domain complexity |
| Addy Osmani | spec.md + task list | Commit-per-task review | Chunked iteration | Any project, granular safety |
| Jesse Vincent | Skills + TDD constraints | Code review between tasks | Subagent dispatch | Projects needing test discipline |
| Freek Van der Herten | Agent-per-task-type | Agent selection + status line | Multi-agent rotation | Ongoing maintenance |

### 2. Parallel-Agent Coordination

#### Simon Willison: /tmp Checkouts and YOLO Mode

**Source:** [simonwillison.net/2025/Oct/5/parallel-coding-agents](https://simonwillison.net/2025/Oct/5/parallel-coding-agents/)

Willison uses fresh checkouts into `/tmp` rather than git worktrees for agent isolation. He runs Claude Code (Sonnet 4.5), Codex CLI (GPT-5-Codex), and Codex Cloud as daily drivers in "YOLO mode (no approvals)" for trusted tasks.

**Four categories that work well in parallel:**
1. Research and proof of concepts
2. Documentation generation
3. Low-stakes maintenance (deprecation warnings, minor irritations)
4. Specified work with detailed specs (reduces review overhead)

**Key insight on the review bottleneck:** "carefully specified and directed actual work" requires less review because goals are pre-determined. The human review cost scales with specification quality, not code volume.

**Security posture:** Acknowledges running agents without sandboxing, noting "the worst that can happen is my source code getting leaked" since most work is open source. Plans to adopt Docker containers for blast radius limitation.

#### Claude Squad: tmux + Worktrees TUI

**Source:** [github.com/smtg-ai/claude-squad](https://github.com/smtg-ai/claude-squad)

Claude Squad provides a terminal UI managing multiple Claude Code, Codex, Gemini, and Aider sessions. Architecture: tmux sessions + git worktrees, each agent on its own branch with its own files and git index. Merge through standard git workflows. Supports split panes for visual monitoring.

#### Native Agent Teams (Anthropic, Feb 2026)

**Source:** [addyosmani.com/blog/claude-code-agent-teams](https://addyosmani.com/blog/claude-code-agent-teams/)

Shipped February 6, 2026 alongside Opus 4.6. Architecture:
- One lead agent orchestrates the operation
- Multiple teammate agents work independently, each in its own context window and git worktree
- Teammates communicate directly with each other via inbox-based messaging (not just reporting to parent)
- Shared task list with dependency tracking and auto-unblocking

**When it works:** Competing debugging hypotheses, parallel specialized code review (security + performance + tests), cross-layer feature work (frontend + backend + tests), research and exploration.

**When it doesn't:** Sequential tasks, same-file edits, extensive dependencies. "Activity doesn't always translate to value." Each teammate is a separate Claude instance multiplying costs.

#### agent_farm: Lock-Based Coordination at Scale

**Source:** [github.com/Dicklesworthstone/claude_code_agent_farm](https://github.com/Dicklesworthstone/claude_code_agent_farm)

Supports 20-50 simultaneous agents with a distributed coordination protocol:
- `active_work_registry.json` -- central registry of all active work
- `agent_locks/` -- individual lock files per agent (`{agent_id}_{timestamp}.lock`)
- `planned_work_queue.json` -- queue for non-started work
- `completed_work_log.json` -- finished tasks log

Each agent generates a unique ID, claims work through the registry, creates lock files for specific files and features, and registers work plans with detailed scope information. Stale lock detection after 2 hours handles abandoned work.

**Monitoring:** Real-time dashboard with context warnings, heartbeat tracking, tmux pane titles. Auto-recovery restarts agents on completion or issues.

**Companion tool (mcp_agent_mail):** Asynchronous coordination layer providing agent identities, inboxes, searchable threads, and advisory file leases via FastMCP + Git + SQLite.

#### Boris Cherny's Approach: Simple Parallelism

**Source:** [getpushtoprod.substack.com/p/how-the-creator-of-claude-code-actually](https://getpushtoprod.substack.com/p/how-the-creator-of-claude-code-actually)

Cherny runs 5 local Claude instances (tabs 1-5, iTerm hotkeys Cmd+1-5) plus 5-10 on claude.ai. Each local session uses its own git checkout (not worktrees). System notifications alert when a Claude needs input. The "teleport" feature pushes sessions between local terminal and web, including starting sessions from the iOS Claude app each morning.

**Key: minimal orchestration infrastructure.** No lock files, no agent farm, no TUI -- just numbered terminal tabs and notifications. Simplicity enables the 259-PRs-in-30-days pace.

### 3. Continuous/Autonomous Workflows

#### Ralph Wiggum Loops (Geoffrey Huntley)

**Source:** [ghuntley.com/ralph](https://ghuntley.com/ralph/), [ghuntley.com/loop](https://ghuntley.com/loop/)

The Ralph loop is deceptively simple: `while :; do cat PROMPT.md | claude-code ; done`. A deterministic development loop that allocates context windows, executes a single task, and repeats. Each iteration gets a fresh context window.

**Economics:** Running Sonnet 4.5 on a Ralph loop costs $10.42/hour. A $50K contract was reportedly completed for $297.

**What works:** Greenfield projects, autonomous development, rapid iteration. Ralph gets projects to ~90% completion before requiring human expertise.

**What breaks:** "There's no way in heck would I use Ralph in an existing code base." Also struggles with multi-agent coordination, which Huntley calls "a red hot mess." Recovery involves git reset or alternative prompts. "You will wake up to a broken code base from time to time."

**Key design decision:** "I don't [plan]. The models know what a compiler is better than I do." The loop is deliberately anti-planning -- trust the model's domain knowledge, fix failures by adding explicit "signs" (instructions) to prevent repeated mistakes.

#### GSD: Structured Autonomy with Fresh Context Per Task

**Source:** [github.com/gsd-build/get-shit-done](https://github.com/gsd-build/get-shit-done)

GSD evolved from a prompt framework to a standalone CLI built on the Pi SDK. Core innovation: every task runs in a fresh 200K-token context window. Task 50 has the same quality as Task 1.

**Architecture:**
- Wave-based parallel execution (independent tasks simultaneous, dependent tasks queued)
- Atomic git commits per task (enables `git bisect`)
- Orchestrator spawns specialized subagents, never does heavy lifting itself
- Orchestrator maintains 30-40% context utilization during entire phase execution

**Six persistent markdown files** maintain state: PROJECT.md, research/, REQUIREMENTS.md, ROADMAP.md, STATE.md, PLAN.md. XML-structured task definitions with embedded verification steps.

**GSD vs. Ralph:** GSD is structured autonomy (discuss-plan-execute-verify-ship with human checkpoints), Ralph is unstructured autonomy (loop until done). GSD prevents context rot by design; Ralph accepts it and uses fresh loops as the mitigation.

#### Realism Assessment

The claims of extended autonomous runs need qualification:
- **Ralph at scale:** No specific evidence of multi-month continuous runs in documented sources. The technique is designed for sustained operation, but Huntley's documented examples focus on specific project completions, not duration marathons.
- **agent_farm:** Designed for unattended operation with auto-recovery, but the coordination complexity (lock files, registries, stale detection) suggests significant babysitting in practice.
- **GSD as CLI:** The most credible autonomous system because it explicitly engineers around context rot (fresh contexts, atomic commits, verification loops).

The honest picture: autonomous operation works for greenfield, well-specified, and structurally repetitive tasks. It breaks on tasks requiring deep contextual understanding of legacy systems, nuanced architectural judgment, or multi-agent state coordination.

### 4. High-Volume Human-in-the-Loop

#### Boris Cherny: 259 PRs in 30 Days

**Source:** [VentureBeat](https://venturebeat.com/technology/the-creator-of-claude-code-just-revealed-his-workflow-and-developers-are), [Pragmatic Engineer](https://newsletter.pragmaticengineer.com/p/building-claude-code-with-boris-cherny)

Key operational details:
- 5 parallel local sessions + 5-10 web sessions
- Opus 4.5 with thinking for everything: "you have to steer it less and it's better at tool use, it is almost always faster than using a smaller model in the end"
- Plan Mode first for most sessions (Shift+Tab twice)
- CLAUDE.md updated multiple times weekly, maintained as shared team document
- GitHub Action for `@claude` in PR reviews that updates CLAUDE.md -- "Human spots issue, Claude updates the rules, future Claude sessions avoid the issue entirely"
- Custom subagents: Code Simplifier, Verify App
- Custom slash command `/commit-push-pr` used daily
- Chrome extension for UI verification, which "2-3x the quality of final results"

**Quality at scale:** The flywheel is the key -- CLAUDE.md captures patterns Claude misses, functioning as institutional knowledge. PR review feedback loops directly into rule updates. This is "compound engineering" -- building intelligence into the codebase itself.

#### Shrivu Shankar: Billions of Tokens/Month at Enterprise Scale

**Source:** [blog.sshh.io/p/how-i-use-every-claude-code-feature](https://blog.sshh.io/p/how-i-use-every-claude-code-feature)

Shankar's team consumes "several billion tokens per month just for codegen" using enterprise API keys (usage-based, not per-seat).

**Operational innovations:**
- 13KB CLAUDE.md with token allocation per internal tool -- "almost like selling 'ad space' to teams," forcing tool teams to simplify documentation
- Only documents tools used by 30%+ of engineers; overflow goes to product-specific files
- "Block-at-Submit" hooks wrapping `git commit` via PreToolUse hooks (test script creates `/tmp/agent-pre-commit-pass` only after successful tests)
- Rejects custom subagents as "a brittle solution" -- prefers "Master-Clone" architecture where the main agent decides when to spawn task clones of itself
- GitHub Action processes customer-triggered PRs from Slack/Jira/CloudWatch alerts
- Runs `query-claude-gha-logs --since 5d | claude -p 'see what the other claudes were getting stuck on and fix it'` to create data-driven improvements
- 2 human reviewers minimum for AI-initiated PRs

### 5. Failure Modes

#### The 70-80% Reliability Ceiling

Multiple sources converge on a fundamental reliability limit for non-trivial AI coding tasks:
- RAND Corporation: 80-90% of AI projects never make it past proof of concept
- 73% of enterprise AI agent deployments experience reliability failures in their first year (Maxim AI)
- Ralph Wiggum gets to ~90% completion on greenfield, then requires human expertise
- GSD exists specifically because "Claude's quality drops as the context window fills up: at 50%+ it starts rushing, at 70%+ hallucinations and forgotten requirements occur"

#### Context Degradation Timeline

**Source:** [morphllm.com/context-rot](https://www.morphllm.com/context-rot), [trychroma.com/research/context-rot](https://www.trychroma.com/research/context-rot)

- **35-minute cliff:** Every AI agent's success rate decreases after 35 minutes of human-equivalent task time. Doubling task duration quadruples the failure rate.
- **Lost-in-the-middle effect (Liu et al., Stanford/TACL 2024):** U-shaped accuracy curve with >30% accuracy drop for information in middle positions. Position 1: ~75% accuracy, positions 5-15: 45-55% accuracy, position 20: ~72% accuracy.
- **Attention dilution:** 10K tokens = 100M pairwise relationships; 100K tokens = 10B; 1M tokens = 1T. Each 10x increase reduces per-token attention weight proportionally.
- **Chroma research (18 LLMs tested):** Performance decline at every context increment, not just near capacity. Model with 200K window shows significant degradation at 50K tokens. Low-similarity needle-question pairs show accelerated decline.
- **Surprising finding:** Models perform better on shuffled, incoherent contexts than logically structured ones, suggesting attention mechanisms process structural patterns counterproductively.

#### Error Compounding in Multi-Step Workflows

**Source:** [Google Research, December 2025](https://research.google/blog/towards-a-science-of-scaling-agent-systems-when-and-why-agent-systems-work/) (180 controlled experiments)

- **17.2x error amplification** in independent multi-agent systems: if a single agent has 5% error rate, independent multi-agent system can reach 86% error rate
- Centralized coordination contains amplification to 4.4x
- Multi-agent benefits depend critically on task structure, not team size
- Self-reinforcing loop: errors -> corrections -> more context -> more noise -> more errors

#### Context Rot Mechanics

**Source:** [MindStudio](https://www.mindstudio.ai/blog/context-rot-ai-coding-agents-explained), [Oracle](https://blogs.oracle.com/developers/agent-memory-why-your-ai-has-amnesia-and-how-to-fix-it)

The degradation loop:
1. Agent gets less accurate as context grows
2. Mistakes require corrections
3. Corrections require reading more files, running more searches
4. Each correction adds noise to context
5. More noise leads to more errors

65% of enterprise AI failures in 2025 were attributed to context drift or memory loss during multi-step reasoning -- not raw context exhaustion.

#### Effective Mitigations

What practitioners actually do to combat these failure modes:

| Mitigation | Used by | Mechanism |
|---|---|---|
| Fresh context per task | GSD, Ralph loops | Restart loop; Task 50 = Task 1 quality |
| Subagent isolation | Shankar, Anthropic (90.2% improvement) | Search traces don't pollute reasoning |
| TDD as guardrail | Reed, Vincent (Superpowers) | Tests catch drift before it compounds |
| Annotation cycles | Tane | Human judgment upfront, not distributed |
| Commit-per-task | Osmani, GSD | Atomic rollback; `git bisect` |
| Status line monitoring | Van der Herten | Visual cue to start fresh at 60%+ |
| CLAUDE.md flywheel | Cherny | Mistakes captured, never repeated |
| Block-at-submit hooks | Shankar | Tests enforced at commit, not mid-planning |
| Plan before execute | All spec-driven practitioners | Prevents wasted cycles on wrong approach |

## Implications for Our Harness

### What we already have that aligns with best practice
- **Tiered routing (T1-T3)** maps to the continuum from simple Ralph-like loops (T1/Fix) to structured spec-driven workflows (T3/Initiative)
- **Checkpoint files + handoff prompts** align with GSD's persistent markdown state and Continuous-Claude's ledgers
- **Specialist agent delegation** matches Cherny's subagent pattern and Vincent's task dispatch
- **beads issue tracking** provides the work coordination that agent_farm achieves through lock files

### Gaps and opportunities
1. **TDD enforcement:** Harper Reed, Jesse Vincent, and Shankar all emphasize TDD as the single best quality signal. We have no TDD hook. Consider a Block-at-Submit hook pattern (Shankar's approach) rather than blocking mid-planning.
2. **CLAUDE.md flywheel:** Cherny's pattern of updating CLAUDE.md from PR review feedback is powerful. We could automate "what did Claude get wrong this session?" capture into CLAUDE.md.
3. **Context health visibility:** Van der Herten's status line (green/yellow/red by context %) would be trivially useful. We have no equivalent.
4. **Fresh context per task for lower tiers:** GSD's insight that Task 50 = Task 1 quality when context is fresh could improve our T1/Fix tier. Consider spawning subagents with fresh context for each fix rather than accumulating in one session.
5. **Annotation cycle support:** Tane's plan.md annotation workflow is a structured human review pattern we don't explicitly support. Our plan steps could include an "annotate plan" checkpoint.

### What to avoid
- **Heavy orchestration infrastructure** (agent_farm's lock files, registries): coordination cost grows fast, and Cherny's minimal approach (5 tabs + notifications) ships more PRs than any orchestration framework
- **Unstructured autonomy** (raw Ralph loops): works for greenfield but "you will wake up to a broken code base" -- our existing structured tiers are the right call
- **Over-investing in multi-agent coordination:** Google's 17.2x error amplification finding suggests centralized coordination (which our harness already does via lead agent + subagents) is strictly better than independent agent swarms for most tasks

## Open Questions

1. **Annotation cycle formalization:** Is there a way to support Tane's inline annotation pattern without adding workflow complexity? Could the plan step in our T2/T3 tiers include explicit annotation checkpoints?
2. **Optimal model selection:** Cherny uses Opus 4.5 exclusively ("faster in the end because you steer it less"). Does this hold for our workloads, or is Sonnet better for isolated subagent tasks?
3. **TDD hook design:** Block-at-submit (Shankar) vs. test-first enforcement (Vincent) -- which pattern fits better with our existing hook infrastructure?
4. **Context health monitoring:** How would we implement a context usage indicator in a terminal-native, foot+tmux environment?
5. **Ralph for T1 fixes:** Would a simple loop pattern work for our Fix tier, where tasks are small and well-specified? What's the minimum viable loop implementation?
